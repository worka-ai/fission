# Why Fission

Fission is shaped by five design choices:

## Deterministic state by default

State updates are driven through explicit action/reducer flow. That makes behavior reproducible and debugging straightforward.

## Clear async boundaries

Long-running work is intentionally separated from rendering via commands, services, and jobs. Background processing can be tested without coupling it directly to UI render steps.

## Cross-platform from one architecture

Whether your target is desktop, web (WASM), iOS, or Android, the model stays consistent. Platform setup is still evolving, but the application model stays stable.

## i18n and accessibility as first-class

Localization and accessibility concerns are included in the core narrative, not left for later. You can add semantic labels, keyboard behavior, and localization early in the first application iterations.

## Worka VM-driven feedback

The same framework logic can be evaluated and iterated inside the browser through a Worka workflow, reducing “edit, compile, switch-target” friction during prototyping and review.
