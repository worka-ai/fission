# Why Fission

Fission is designed around five practical differentiators that show up directly in day-to-day development.

## 1) Deterministic state by default

State changes are explicit and reproducible. Actions describe intent and reducers apply a small, bounded state transition. This improves traceability in code review and debugging.

## 2) Explicit async boundaries

UI builds stay focused on rendering decisions, while asynchronous behavior is represented through commands, services, and jobs. You can keep business logic and external integration paths separate and testable.

## 3) One architecture for multiple targets

The same app model is intended to span desktop, web, and mobile builds. You still need target-specific packaging, but behavior and structure remain shared.

## 4) Production-quality baseline concerns

i18n and accessibility are part of the architecture. That means localization keys, semantics, and focus behavior are introduced early, not retrofitted at release time.

## 5) Fast feedback with browser-backed preview

The Worka VM-backed workflow lets teams validate behavior quickly before full packaging. That improves collaboration and reduces time spent iterating on layout, behavior, and async timing.

## What this means in practice

If you are shipping a real product team, the payoff is less context switching and a shorter path from idea → validation → implementation.
