# Why Fission

You do not need to adopt every feature at once. You need a framework shape that makes it easier to scale from first screen to production flow.

**Deterministic behavior first**
Actions, reducers, and state transitions are explicit. That gives you one path from user input to state change and one place to test it.

**Clear async boundaries**
Commands, services, and jobs split effectful work from pure rendering. UI code stays readable while integration details remain controlled and testable.

**One app model across targets**
The same core model is used for desktop and web/WASM targets. You still define target build steps, but your architecture does not fork.

**Global quality in the base model**
Localization and accessibility are treated as first-order concerns. Keys, labels, focus behavior, and contrast are part of your normal implementation flow.

**Fast feedback loop for visual and behavior changes**
A browser-based preview loop keeps edits visible quickly while the app model is still stable.

This structure helps teams ship confidently because the same patterns repeat from first button to first release.
