# State and actions

State gives you a stable contract. Actions express intent. Reducers apply state changes.

## Practical model

Use domain-specific action names (`TodoToggled`, `ThemeChanged`, `ItemDeleted`) and keep reducer functions small.

```rust
#[derive(Action, Serialize, Deserialize, Debug, Clone, PartialEq, Eq)]
struct TodoToggled {
    id: usize,
}

fn reduce_todo_toggled(state: &mut AppState, action: TodoToggled, _ctx: &mut ReducerContext<AppState>) {
    if let Some(todo) = state.todos.get_mut(action.id) {
        todo.done = !todo.done;
    }
}
```

## Team-level guidance

- reducers are pure,
- no async calls in reducer bodies,
- one reducer should do one concept,
- tests assert both positive and negative transitions.

When action naming is clean, maintenance becomes much cheaper later.
