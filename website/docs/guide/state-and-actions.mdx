# State and actions

For someone new to Fission, the state model is the single best place to start. Your app state is the source of truth.

A good action uses product meaning (`TodoToggled`, `ThemeChanged`) instead of generic names. A reducer takes an action and updates state deterministically.

```rust
#[derive(Action, serde::Serialize, serde::Deserialize, Debug, Clone, PartialEq, Eq)]
struct TodoToggled {
    id: usize,
}

fn reduce_todo_toggled(
    state: &mut TodoState,
    action: TodoToggled,
    _ctx: &mut ReducerContext<TodoState>,
) {
    if let Some(todo) = state.todos.get_mut(action.id) {
        todo.done = !todo.done;
    }
}
```

Keep these habits until your reducer paths are stable:

- action names should match domain intent,
- each reducer should do one conceptual update,
- avoid side effects in reducers,
- test each transition before adding more behavior.

From here, use [Commands, services, and jobs](/docs/guide/commands-services-jobs) when behavior talks to the outside world.
