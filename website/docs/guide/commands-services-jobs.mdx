# Commands, services, and jobs

Use the three-way split only when work must leave the pure UI/reducer flow.

A command handles one-shot async work. A service is reusable and long-lived. A job handles repeating or cancellable work tied to lifecycle events.

```text
UI action -> command/service/job -> completion action -> reducer -> render
```

Choose by responsibility:

- one response is needed once: use a command,
- shared integration surface is needed: use a service,
- ongoing or cancellable behavior is needed: use a job.

This keeps behavior predictable while still allowing rich integrations.
