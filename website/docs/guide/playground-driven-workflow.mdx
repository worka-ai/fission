# Playground-driven workflow

`playground` workflow for practical quality checks.
This section shows the practical flow you use to validate behavior before expanding to more output targets.

Edit your generated app locally, then run a target script for the platform you need to verify.
This gives real feedback from:

- desktop (`cargo run`)
- web (`./platforms/web/run-browser.sh`)
- iOS/Android (`./platforms/ios/run-sim.sh`, `./platforms/android/run-emulator.sh`)

Use one small behavior change, then re-run one target.

From a generated app directory:

- Desktop loop (builds dependencies and launches the app):

```bash
cargo run
```

- Browser loop (requires `cargo fission add-target web`, then `wasm32` toolchain + wasm-pack):
  - `rustup target add wasm32-unknown-unknown`
  - `cargo install wasm-pack`

```bash
cargo fission add-target web && ./platforms/web/run-browser.sh
```

This command only works from the generated app directory with the generated `platforms/web` scripts present.

- iOS and Android loops use generated run scripts from the same pattern.

From repo root (checked-in smoke app):

```bash
./examples/web-smoke/platforms/web/run-browser.sh
```

This prints the same localhost host and serves `examples/web-smoke`.

Use this as your quality checkpoint for every run:

- visible state transition in the UI,
- action behavior matches your expected state change,
- keyboard focus lands on controls in expected order,
- button and action labels remain readable and meaningful.

When you are ready for deeper mobile smoke verification, use the scripts and target notes in the [CLI-and-targets guide](https://github.com/worka-ai/fission/blob/main/docs/cli-and-targets.md), [platform smoke tests](https://github.com/worka-ai/fission/blob/main/docs/platform-smoke-tests.md), and generated platform notes under [fission-cli](https://github.com/worka-ai/fission/blob/main/crates/tools/fission-cli/README.md).
