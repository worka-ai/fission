### Platform-first order and caveats

Fission is desktop-first. You should get desktop behavior working first, then add generated host targets for wider coverage.

<Tabs>
<TabItem value="generated" label="Generated app" default>

Start from a fresh scaffold, verify desktop, then add targets as needed:

```bash
fission init my-fission-app
cd my-fission-app
cargo run
cargo fission add-target web ios android
```

Run host scripts from the **generated app root**:

```bash
./platforms/web/run-browser.sh      # requires wasm32-unknown-unknown + wasm-pack
./platforms/ios/run-sim.sh          # requires Xcode simulator SDK + platforms/ios/README.md
./platforms/android/run-emulator.sh  # requires SDK/NDK/toolchain + platforms/android/README.md
```

Do not assume generated targets behave identically out of the box:

- web requires `wasm32-unknown-unknown` and `wasm-pack`,
- iOS requires simulator toolchain prerequisites,
- Android requires a valid Android SDK/NDK and target setup.

</TabItem>

<TabItem value="repo" label="Repo examples">
Use the checked-in examples when you need deterministic host files from the repository:

```bash
./examples/web-smoke/platforms/web/run-browser.sh
./examples/mobile-smoke/platforms/ios/run-sim.sh
./examples/mobile-smoke/platforms/android/run-emulator.sh
```

These are useful for baseline verification and smoke checks, but the normal product flow is generated per app.
When you need host output parity, compare both flows before release verification.
</TabItem>
