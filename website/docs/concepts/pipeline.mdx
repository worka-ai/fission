# Rendering pipeline

The frame pipeline is split into clear stages so debugging stays practical.

Your tree is built from state, then lowered into an internal representation, then laid out, then painted, and finally rendered.

A stable mental model:

1. build widget structure,
2. lower to render graph,
3. compute layout,
4. generate paint commands,
5. render output.

You do not need to memorize internals yet. You need to know where a change belongs. If visuals are wrong, check each stage in order and stop as soon as the mismatch disappears.

This is why the docs introduce state and layout before platform tooling: once your mental model is stable, pipeline issues become mechanical rather than mysterious.
