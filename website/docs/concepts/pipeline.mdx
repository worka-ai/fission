# Rendering pipeline

Use this exact sequence while debugging:

- build Node tree
- lower Node tree to `CoreIR`
- compute `LayoutSnapshot`
- build/cache `DisplayList` or `RenderScene`
- render through the active backend

That sequence is what most regressions follow. If visuals are wrong, inspect each stage in order and stop when the mismatch first appears:

- Node tree content changed unexpectedly -> check state, reducer, and build logic
- CoreIR or semantics changed unexpectedly -> check branching and `Semantics`
- `LayoutSnapshot` changed unexpectedly -> check constraints and layout inputs
- `DisplayList`/`RenderScene` output changed unexpectedly -> check style changes and cache invalidation
- backend output changed unexpectedly -> check runtime/backend flags and host config

## Where diagnostics hook in

Diagnostics are enabled through the `FISSION_DIAG` environment variable (for example `FISSION_DIAG=input,layout`).
Use `FISSION_DIAG_LEVEL=debug` to enable the debug-level events described below.
That gives per-stage tracing at these pipeline hooks:

- `input`: action dispatch, reducer input, and pointer/gesture events
- `layout`: snapshot generation, incremental diffs, and paint-cache summaries (`PaintSummary`)
- `frame`: frame start/end lifecycle events around the render loop

The `test` category is defined but currently has no emitters.
