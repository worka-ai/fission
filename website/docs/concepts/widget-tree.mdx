# Widget tree model

Think of the widget tree as the deterministic UI projection of current state.
For first-time users, this keeps debug work practical: your state changes, then `build` renders those changes.

Use this exact source flow whenever behavior is unclear:

- `state -> action -> reducer -> state -> Widget::build -> Node -> CoreIR -> layout -> paint -> backend`

One compact diagram-like list:

- state
- action dispatch
- reducer
- new state
- `Widget::build`
- `Node`
- `CoreIR`
- `LayoutSnapshot`
- `DisplayList` / `RenderScene`
- active backend

For first-time users, this is the first place to look:

- `Widget::build` is a pure projection from state.
- `CoreIR` is what the runtime diffs frame-to-frame, using `diff_ir(prev_ir, next_ir)`.
- `CoreIR` is the typed intermediate structure used by layout and paint.
- if visuals diverge, isolate the first stage where expectation changes.

If your first build is getting stuck, use this quick checklist:

- start from initial state defaults
- dispatch one action (one interaction)
- confirm reducer output before looking at UI
- inspect `CoreIR` shape before checking paint
- then inspect `LayoutSnapshot` and display-list output

This keeps you anchored to a small, testable flow instead of overfitting on one render detail.
