# Widget tree model

Think of the widget tree as the blueprint for one rendered frame. It is a typed description of what should be shown right now.

If you have not seen Fission before, treat the tree like the source of truth. Your app state goes in, and the tree is the framework’s plan for how to draw it.

The main benefit is traceability.

- same state produces a stable tree,
- every behavior change starts as a tree change,
- layout and paint problems are easier to isolate because the render path is explicit.

Use this mindset while building:

- keep `build` pure,
- keep branch decisions small,
- make each widget represent one meaningful screen decision.

When behavior feels hard to reason about, trace from interaction to action, from action to reducer, then from reducer result to updated tree.
