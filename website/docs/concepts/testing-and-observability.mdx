# Testing and observability

Fission is most useful when teams can trust behavior after changes. The order matters: validate the smallest unit first, then expand.

Start with reducer tests because they are closest to your source of truth. Then verify command and service behavior, then run meaningful UI flows.

A practical sequence:

- reducer tests for state transitions,
- mocked command/service tests,
- targeted playground checks for interaction,
- target smoke checks before release.

Keep logs and transition data structured. If a behavior regresses, you should be able to answer what action happened, what reducer executed, and what async path produced the result.

Do not wait for a production bug to add this flow. Add it while features are still small.
