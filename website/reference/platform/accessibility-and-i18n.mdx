# Accessibility and i18n

These two surfaces are separate in code, but they often ship together.

## Accessibility

Semantics are produced by the runtime from widget lowering and layout. Shells then bridge the resulting semantic tree into native accessibility APIs.

Important consequences:

- labels and roles should come from the shared tree, not per-platform code,
- live tests can inspect semantic output before or alongside native tooling,
- overlay and focus behavior should be validated through semantics, not only screenshots.

## I18n

Translated text enters through `Env.i18n` and `Env.locale`, then resolves during build/layout.

That means locale changes can affect:

- visible text,
- semantic labels,
- layout width and overflow,
- focus order if dynamic content changes the tree.
