# State system

The state system is the place where behavior is decided.

- `AppState` defines the root state type,
- `Action` describes user or system intent,
- `Reducer` transforms state deterministically,
- `ReducerContext` carries runtime metadata.

The flow is intentionally simple:

1. action dispatch,
2. reducer receives `&mut` state,
3. update state,
4. next build recomputes UI.

Keep reducers narrow and testable, then grow behavior via commands and services only when needed.

If multiple parts of your app mutate shared state, start by splitting domain actions first.
