# State system

Fission keeps product logic in a small set of public contracts.

## Primary types

- `AppState`: your durable product model
- `Action`: a serializable intent type with a stable ID
- `ActionEnvelope`: erased transport form used by the runtime
- `Handler<S, A>`: the canonical reducer signature
- `ReducerContext<S>`: effect builder plus action input metadata
- `ActionRegistry<S>`: a per-frame or startup-time reducer registry

## Reducer shapes

Fission accepts both legacy two-argument reducers and the modern three-argument form, but the three-argument form is the important one because it gives access to `ReducerContext`.

```rust
fn handle(state: &mut MyState, action: MyAction, ctx: &mut ReducerContext<MyState>)
```

## Binding reducers

The two normal binding paths are:

- `ctx.bind(action, handler)` inside `build()` for local UI handlers
- `ActionRegistry::register(handler)` plus shell `absorb_registry(...)` for global startup wiring

## Selectors

Use `Selector<S>` when you want a stable, testable derived view model without mutating app state during build.

The `counter` example's `CounterVM` is the canonical pattern in this repo.

## Why the model matters

Because actions are typed and reducers are explicit, the same transition can be:

- replayed in tests,
- driven by mouse, keyboard, IME, or accessibility input,
- observed through diagnostics,
- reused across desktop, web, and mobile hosts.
