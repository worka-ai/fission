# Commands, services, and jobs

The reducer-side async API lives on `ReducerContext::effects`.

## Job

Use `app(job, request)` for one-shot work with a single success or failure path.

Important types:

- `JobSpec`
- `JobRef<J>`
- `JobCtx`

## Service

Use `start_service(slot, config)` for long-lived work that may emit events or accept later commands.

Important types:

- `ServiceSpec`
- `ServiceSlot<Svc>`
- `ServiceCtx`
- `ServiceBindings`

## Command

Use `command(slot, payload)` to talk to an already-running service.

## Stop

Use `stop_service(slot)` when the service lifecycle should end explicitly rather than by resource reconciliation.

## Common callback hooks

The effect builders support `on_ok(...)`, `on_err(...)`, and service-specific bindings such as:

- `on_started(...)`
- `on_start_failed(...)`
- `on_event(...)`
- `on_stopped(...)`
- `on_command_ok(...)`
- `on_command_err(...)`

## Shell registration

Jobs, services, and custom capabilities are registered at app startup through shell `with_async(...)`.

That keeps runtime plumbing centralized and visible to code review.
