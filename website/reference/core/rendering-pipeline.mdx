# Rendering pipeline

The public surface is authored as widgets, but the runtime works in a stricter staged pipeline.

## Stages

1. `Widget::build()` returns a `Node` tree.
2. the tree is lowered into Core IR.
3. layout resolves constraints into a `LayoutSnapshot`.
4. paint compilation emits a display list.
5. the active shell presents through its renderer backend.

## Public types worth knowing

- `Node`, `CustomNode`, and `LowerDyn` for authoring
- `LoweringContext` and `NodeBuilder` for custom lowering
- `CoreIR`, `NodeId`, and `WidgetNodeId` for runtime identity
- `LayoutSnapshot`, `LayoutRect`, and `LayoutSize` for geometry

## Why this page exists

The most useful debugging question is often "which stage is wrong first?"

- wrong branch or text -> state or build
- wrong geometry -> layout
- right layout but wrong pixels -> paint/theme/media
- launch or input mismatch -> shell/runtime boundary

## Custom rendering hooks

Use `LowerDyn` when a widget needs direct IR control. The editor and 3D examples are the best repo-backed references for that escape hatch.
