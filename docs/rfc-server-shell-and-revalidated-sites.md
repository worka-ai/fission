# RFC: Server Shell, Revalidated Sites, Workers, and WASM Islands

Status: proposal  
Audience: Fission shell, static-site, server, cache, web, CLI, and application authors  
Scope: using Fission to build server-backed websites and web applications that combine static rendering, server rendering, progressive enhancement, and targeted WASM islands

## 1. Summary

Fission should support server-side web applications without replacing Rust web servers. The server shell should let an Axum, Actix, Hyper, or other Rust HTTP service route requests into a real Fission application, render Fission widget trees to HTML, cache rendered output, rebuild expired pages without recompiling the app, and attach browser-side behavior only where the page explicitly needs it.

The model is a hybrid of four execution tiers:

1. revalidated pages, rendered from Fission widget trees into HTML and cached with TTLs, tags, and explicit invalidation;
2. request-rendered pages, rendered from Fission widget trees on demand for private, session-specific, or highly dynamic content;
3. progressive-enhancement workers, compiled from Rust to small WASM workers that send DOM/window/document commands to a generated main-thread bridge; and
4. WASM islands, compiled from Rust to targeted interactive Fission browser modules for areas that need full Fission action/reducer behavior in the browser.

The same route can mix these tiers. An ecommerce category page can be revalidated every few minutes, render product data through server-side jobs, use a worker for the mobile navigation drawer and client filtering controls, and mount a WASM island for a product customizer or cart drawer.

The important constraint is that static, revalidated, and server-rendered pages are still Fission widget trees. The server shell must not introduce an unrelated HTML template system, a second router, or a client-controlled server component model. Jobs, services, actions, reducers, widgets, routing, design-system tokens, and capabilities remain the application model.

## 2. Goals

- Render real Fission routes to HTML at build time and request time.
- Let rendered pages expire and be rebuilt by a running server without recompilation or redeployment.
- Support cache eviction, TTLs, tag invalidation, stale-while-revalidate, and explicit route invalidation.
- Provide a `Cache` trait with Moka as the default in-memory implementation and room for Redis, database, CDN, and layered cache implementations.
- Allow cache pipelines such as in-memory Moka first, Redis second, and object storage later.
- Keep Fission's existing jobs, services, actions, reducers, and widget model instead of adding a parallel server-component model.
- Support small Rust-authored progressive-enhancement workers for DOM/CSS/browser behavior where a full Fission island is too heavy.
- Support one WASM artifact per declared island and one WASM artifact per declared worker where practical.
- Generate the main-thread JavaScript bridge needed for worker-driven DOM updates.
- Keep primary content present in HTML before JavaScript or WASM runs.
- Keep browser code opt-in and route-local.
- Provide clear security boundaries so clients cannot choose arbitrary Rust functions, jobs, services, widgets, or files to execute on the server.
- Provide a path for ecommerce, documentation, dashboards, account portals, forms, search, and authenticated applications.

## 3. Non-goals

- Building a new Rust web server.
- Replacing Axum, Actix, Hyper, Tower, or equivalent HTTP infrastructure.
- Replacing Fission's existing static-site target.
- Shipping every route as a browser WASM app.
- Allowing the browser to invoke arbitrary server-side Rust code by name.
- Introducing a server-component-style execution model.
- Requiring JavaScript for primary page content.
- Requiring Redis, a CDN, or any hosted cache service for basic operation.
- Making the hot rendering path depend on JSON serialization.
- Making all pages dynamic by default.

## 4. Terminology

Revalidated page means a route rendered from a Fission widget tree whose HTML can be cached and regenerated by the running server after a TTL expires or an invalidation event occurs.

Request-rendered page means a route rendered from a Fission widget tree for the current request, session, user, locale, permissions, or other private context. It may use private caches internally, but the full HTML response is not publicly reusable unless configured to be safe.

Progressive-enhancement worker means a small Rust-to-WASM Web Worker that handles browser-side behavior and sends DOM/window/document operations to a generated main-thread bridge. The worker does not render Fission UI into a canvas and does not directly own a Fission display list.

WASM island means a targeted browser-side Fission runtime mounted into a specific page region. It can run Fission action/reducer logic, render with the web renderer, and manage an interactive area independently of the rest of the page.

Main-thread bridge means the generated JavaScript module loaded by the page that receives worker messages and applies allowed DOM, CSS, history, focus, scroll, storage, or document operations. Workers cannot manipulate the DOM directly, so this bridge is mandatory for worker-driven progressive enhancement.

Cache tag means a stable invalidation label such as `product:shoe-123`, `category:trainers`, `account:42`, or `site-nav`. A cached page, fragment, or job result can carry many tags.

Cache pipeline means an ordered set of cache layers. For example, Moka can serve hot entries in memory and fall through to Redis for shared multi-process entries.

## 5. Architecture overview

The server shell is an adapter between an HTTP server and a Fission application.

```text
HTTP server
  -> FissionServerService
      -> route resolution
      -> request Env construction
      -> session and capability context
      -> cache lookup
      -> Widget::build
      -> job drain for blocking server data
      -> reducer/action dispatch when handling form/action requests
      -> Node/Core IR/Semantics to HTML/CSS/assets
      -> worker and island manifest emission
      -> cache write
  -> HTTP response
```

The static-site shell and server shell should share the HTML renderer, CSS extraction, asset manifest, metadata generation, Markdown/content rendering, link validation, and island/worker declaration machinery. They differ in when rendering happens and how cache state is stored:

| Target | Rendering time | Cache scope | Rebuild behavior |
|---|---|---|---|
| static site build | build time | filesystem/CDN/browser | requires a new build unless paired with server revalidation |
| server revalidated page | request time on miss/expiry/invalidation | server cache and optional external cache | rebuilt by running server without recompilation |
| request-rendered page | every request or private cache miss | request/session/user cache | rebuilt for the request context |
| browser island | browser runtime | browser memory/storage where configured | rebuilt by browser-side Fission runtime |
| worker enhancement | browser runtime | browser memory/storage where configured | runs command-style DOM operations |

## 6. Route model

Server-capable routes use the existing Fission router/widget model. The server shell needs route metadata so it can decide how a route is rendered and cached.

```rust
pub enum WebRouteMode {
    Static,
    Revalidated(RevalidationPolicy),
    Server(ServerRenderPolicy),
    ServerPrivate(ServerPrivatePolicy),
    ClientApp(ClientAppPolicy),
}

pub struct WebRoute {
    pub path: RoutePattern,
    pub mode: WebRouteMode,
    pub metadata: RouteMetadata,
    pub entry: RouteEntry,
}
```

`Static` means the route can be rendered by `fission site build` and emitted as normal files. It has no server rebuild behavior on its own.

`Revalidated` means the route can be rendered by `fission site build` for the initial artifact and also rebuilt by the server when stale, evicted, or invalidated.

`Server` means the route is public but depends on request-time data. It can still use public caches if its cache key is complete.

`ServerPrivate` means the route is specific to a session, account, permission set, tenant, locale, cart, or other private context. It must not be stored in a public cache.

`ClientApp` means the route intentionally mounts a full browser Fission app. This is for app-like surfaces, not normal content pages.

The route mode is not a different authoring model. Each route still renders a Fission widget tree through the same router and `Widget::build` path.

## 7. Revalidation and page lifecycle

A revalidated page does not live forever. It is a cached render result with an expiry policy.

```rust
RevalidationPolicy::new()
    .ttl(Duration::from_secs(300))
    .stale_while_revalidate(Duration::from_secs(60))
    .tags(|params| vec![format!("product:{}", params["slug"]), "catalog".into()])
```

Request flow for a revalidated route:

1. Resolve the route and request context.
2. Build the page cache key from route path, route params, normalized query keys, locale, selected theme, design-system fingerprint, app build id, and any configured public variation fields.
3. Ask the cache pipeline for the full-page entry.
4. If the entry is fresh, return it.
5. If the entry is stale and stale serving is allowed, return stale HTML and schedule background regeneration.
6. If the entry is missing, expired beyond the stale window, or explicitly invalidated, rebuild synchronously unless the route policy allows serving a fallback.
7. During rebuild, execute the Fission route, drain blocking jobs, render HTML/CSS metadata, and write the cache entry with its TTL and tags.
8. Return the rebuilt response.

This means an app can deploy once and keep rebuilding category pages, product pages, documentation pages, campaign pages, and landing pages as data changes.

A pure static host cannot perform server revalidation by itself. If the artifact is deployed to static hosting only, TTLs affect HTTP cache headers and generated metadata, but page regeneration requires a new build or an external serverless/runtime integration. The server shell is the component that enables rebuild without redeployment.

## 8. Cache model

### 8.1 Cache trait

The server shell must depend on a cache trait, not a concrete cache store.

```rust
pub trait Cache: Send + Sync + 'static {
    fn get(&self, key: &CacheKey) -> CacheFuture<Option<CacheEntry>>;
    fn put(&self, entry: CacheEntry) -> CacheFuture<()>;
    fn remove(&self, key: &CacheKey) -> CacheFuture<()>;
    fn invalidate_tag(&self, tag: &CacheTag) -> CacheFuture<InvalidationReport>;
    fn invalidate_tags(&self, tags: &[CacheTag]) -> CacheFuture<InvalidationReport>;
    fn contains_fresh(&self, key: &CacheKey, now: Timestamp) -> CacheFuture<bool>;
}
```

The exact async type can use `async_trait`, boxed futures, or associated future types. The API requirement is that it supports lookup, insertion, key removal, tag invalidation, and freshness checks. Implementations may support additional introspection and metrics.

### 8.2 Cache entries

```rust
pub struct CacheEntry {
    pub key: CacheKey,
    pub value: CacheValue,
    pub scope: CacheScope,
    pub created_at: Timestamp,
    pub fresh_until: Timestamp,
    pub stale_until: Option<Timestamp>,
    pub tags: Vec<CacheTag>,
    pub vary: CacheVary,
    pub content_hash: ContentHash,
    pub metadata: CacheMetadata,
}

pub enum CacheValue {
    FullPage(RenderedPage),
    Fragment(RenderedFragment),
    JobResult(StoredJobResult),
    AssetMetadata(AssetMetadata),
}
```

The cache must be able to store full pages, fragments, and job results. These are related but not identical lifecycles. A product details page may be cached for five minutes, its product summary job result may be cached for one minute, and the recommendation fragment may be private per user and uncached.

### 8.3 Cache scopes

```rust
pub enum CacheScope {
    Public,
    PrivateSession,
    PrivateUser,
    PrivateTenant,
    NoStore,
}
```

`Public` entries can be reused across users when the cache key and vary fields are complete.

`PrivateSession` entries are bound to one anonymous or authenticated session.

`PrivateUser` entries are bound to one authenticated user.

`PrivateTenant` entries are bound to one tenant/workspace/account boundary.

`NoStore` entries must not be written to persistent cache.

The server shell must prevent a route marked private from being inserted into a public cache layer. If a route uses private request data, session values, authentication state, or user permissions while declaring a public cache policy, validation should fail unless the route explicitly proves that those inputs are not used in rendered output.

### 8.4 Default implementation: Moka

Fission should ship a Moka-backed in-memory cache as the default development and single-process production cache.

Moka is the right default because it provides high-performance in-memory caching, TTL support, size-based eviction, and async-friendly usage without requiring external infrastructure.

Default behavior:

```toml
[server.cache]
provider = "moka"
max_capacity = 10000
ttl = "5m"
stale_while_revalidate = "1m"
```

The Moka implementation should support:

- full-page cache entries;
- fragment entries;
- job result entries;
- TTL and stale windows;
- capacity eviction;
- tag index maintenance;
- metrics for hit, miss, stale hit, eviction, invalidation, and rebuild duration.

Moka is process-local. It is sufficient for local development, simple deployments, and single-server applications. Multi-process or multi-region deployments need a shared cache implementation or a pipeline.

### 8.5 Redis and future stores

Redis should be the first shared external cache implementation.

```toml
[server.cache]
provider = "redis"
url_env = "FISSION_REDIS_URL"
prefix = "storefront:prod"
default_ttl = "5m"
stale_while_revalidate = "1m"
```

The Redis implementation should support the same logical API as Moka. It should store cache values in a deterministic binary format owned by Fission, not JSON. Tag indexes can be maintained as Redis sets or another structure chosen by the implementation. The implementation must avoid unbounded per-tag scans where possible.

Future stores may include database-backed caches, object storage-backed page caches, CDN purge adapters, and custom user-provided implementations. Those should implement the same `Cache` trait or a compatible extension trait.

### 8.6 Cache pipelines

A cache pipeline lets one layer overflow or fall through to another.

```toml
[[server.cache.layers]]
name = "hot"
provider = "moka"
max_capacity = 50000
ttl = "2m"

[[server.cache.layers]]
name = "shared"
provider = "redis"
url_env = "FISSION_REDIS_URL"
ttl = "10m"
```

Read behavior:

1. Ask layer 0.
2. If layer 0 misses, ask layer 1.
3. If layer 1 hits, promote the entry back to layer 0 if allowed.
4. If every layer misses, render/rebuild.

Write behavior:

- Write through to all configured writable layers by default.
- Allow route policies to mark entries as hot-only, shared-only, or no-store.
- Never write private entries to a public/shared layer unless the layer is configured as private and the key includes the required privacy boundary.

Invalidation behavior:

- `invalidate_tag` must be sent to every layer that may contain entries for that tag.
- A layer that cannot perform tag invalidation must declare that limitation during readiness checks.
- Partial invalidation failure must be reported and should make deployment readiness fail for routes that require strong invalidation.

### 8.7 Cache keys

Cache keys must be deterministic, compact, and safe to compare across layers.

Inputs for full-page cache keys should include:

- route id;
- normalized path params;
- allowed query params;
- locale;
- public theme/design-system variant where output differs;
- app build id;
- renderer version;
- design-system fingerprint;
- content collection version where applicable;
- explicitly declared `vary` fields.

Private cache keys also include session, user, tenant, permission, or cart identifiers as required by the scope.

The server shell must not implicitly vary on every header or cookie. Routes declare their allowed vary fields so accidental cache explosion is avoided.

## 9. Data fetching with jobs

Server-side data fetching should use Fission jobs. A route does not need a separate `loader` concept.

A job used during server rendering can declare a server resource policy:

```rust
pub enum ServerResourcePolicy {
    Blocking,
    Deferred,
    IslandOnly,
    NoServerExecution,
}
```

`Blocking` means the server render waits for the job result before producing the final HTML. This is appropriate for product details, article content, prices that must be visible at first paint, checkout forms, and account pages.

`Deferred` means the server emits a placeholder and a worker or island can fetch/update after first paint. This is appropriate for recommendations, analytics summaries, low-priority personalization, or sections that can appear later.

`IslandOnly` means the job belongs to a browser island and must not run during server rendering.

`NoServerExecution` means the job is not available in the server environment.

Server rendering flow for jobs:

1. Build the route widget tree.
2. Collect job resources requested by widgets such as `FutureBuilder`.
3. For each blocking job, compute its cache key.
4. Check the job cache.
5. Execute the job if needed.
6. Dispatch success or failure actions into the route state.
7. Rebuild the widget tree with resolved data.
8. Repeat until no new blocking jobs are pending or a configured pass limit is reached.
9. Render HTML.

The pass limit prevents accidental infinite fetch/rebuild loops. If the pass limit is exceeded, rendering fails with a diagnostic that identifies the route and job chain.

## 10. Services on the server

Services remain the model for long-running, streaming, shared, or lifecycle-managed work.

Server-side services are appropriate for:

- live inventory feeds;
- order-status streams;
- notifications;
- websocket fan-out;
- payment webhook processing;
- cache warming;
- background synchronization;
- search index updates;
- analytics ingestion.

A route can subscribe to service state through normal Fission mechanisms. During server rendering, service reads must be deterministic with respect to the request and cache policy. If service state is volatile and not captured in the cache key or tags, the route must be private, no-store, or explicitly revalidated with a short TTL.

Services may emit cache invalidation events:

```rust
ctx.cache.invalidate_tag("product:shoe-123");
ctx.cache.invalidate_tag("inventory:shoe-123");
ctx.cache.invalidate_route("/products/shoe-123");
```

The invalidation API is part of the server context, not a browser capability. A browser action may trigger a reducer/job/service that leads to invalidation, but the client does not directly invalidate arbitrary tags.

## 11. Actions and reducers over HTTP

Server-side mutations should use Fission actions and reducers. A route does not need a separate `server_action` concept.

For HTML forms and command buttons, the server shell emits action endpoints and signed action tokens.

Flow for a POST action:

1. Browser submits a form or fetch request to a Fission server action endpoint.
2. Server verifies route, session, CSRF token, origin policy, and signed action token.
3. Server deserializes the action payload using the action type registered by the compiled app.
4. Server dispatches the action through the reducer path for the route/session/app state.
5. Reducer updates state and may trigger jobs, services, cache invalidation, redirects, or response fragments.
6. Server persists session state if needed.
7. Server returns a redirect, a full page, a fragment patch, or an island/worker message response depending on the request type.

The browser never sends a Rust function name, module path, trait impl, job name, widget name, or arbitrary code identifier to execute. It sends a token generated by the server for a specific route, action, and allowed payload shape.

## 12. Progressive-enhancement workers

Progressive enhancement covers behavior that needs DOM/CSS/window/document manipulation but does not need a full Fission-rendered island.

Good worker use cases:

- responsive navigation drawer;
- collapsible documentation sidebars;
- tab panels;
- accordions;
- theme toggle;
- client-side search UI;
- filter controls over already-rendered content;
- copy-to-clipboard buttons;
- small form affordances;
- scrollspy and table-of-contents updates;
- non-critical analytics events;
- client-side validation hints.

A worker runs Rust compiled to WASM in a Web Worker. It cannot access the DOM directly. The generated main-thread bridge owns DOM access and applies a restricted set of operations sent by the worker.

### 12.1 Worker declaration

A widget can declare that an HTML-rendered subtree should be enhanced by a worker.

```rust
HeaderNav::new()
    .enhance_with(HeaderNavWorker::new())

DocsSidebar::new(items)
    .enhance_with(CollapsibleSidebarWorker::new())
```

The declaration means:

- render normal HTML for the subtree;
- assign stable node ids for event targets and controlled nodes;
- compile the worker entry into a route-local or shared worker WASM artifact;
- load the generated main-thread bridge only on pages using workers;
- send boot data and events to the worker;
- apply the worker's allowed DOM operations.

The worker should not be required for reading the page. If the worker fails to load, the base HTML remains usable.

### 12.2 Worker build output

```text
target/fission/site/
  assets/workers/header-nav.<hash>.wasm
  assets/workers/docs-sidebar.<hash>.wasm
  assets/fission-worker-bridge.<hash>.js
  assets/worker-manifest.<hash>.json
```

The build should produce one worker artifact per worker entry where practical. The CLI may deduplicate identical workers shared across routes. For very small deployments, a configuration option may bundle workers, but separate artifacts are the default target because they reduce unnecessary download size.

## 13. Worker message protocol

Fission should own a small, versioned structured-clone protocol. Existing generic RPC protocols are either too JavaScript-shaped, too dynamic, too JSON-heavy, or not DOM-patch-oriented enough for Fission's needs.

### 13.1 Transport

- Transport is `postMessage` between the main thread and worker.
- Values use structured clone-compatible data.
- The default protocol must not require `JSON.stringify` or `JSON.parse` on the hot path.
- Messages carry a protocol version and worker instance id.
- DOM operations are batched.
- The bridge must reject unknown operation types for the negotiated protocol version.

### 13.2 Main to worker

```rust
pub enum MainToWorker {
    Boot(WorkerBoot),
    Event(WorkerDomEvent),
    Resize(WorkerResize),
    VisibilityChanged { visible: bool },
    ThemeChanged { theme_id: String },
    LocaleChanged { locale: String },
    Response(WorkerResponse),
    Shutdown,
}
```

`Boot` includes route id, base URL, root node id, initial page state, locale, theme, feature flags, and any server-emitted worker props.

`Event` includes target node id, event kind, selected event fields, input value where relevant, keyboard modifiers, pointer metadata where relevant, and a sequence number.

`Response` carries the result of a worker request such as reading storage, asking for clipboard permission, fetching server data, or querying document state.

### 13.3 Worker to main

```rust
pub enum WorkerToMain {
    Ready,
    DomBatch(DomBatch),
    Request(WorkerRequest),
    Navigate(NavigateRequest),
    Log(WorkerLog),
    Error(WorkerError),
}
```

`DomBatch` is the primary output. It contains a sequence number, optional transaction id, and a list of allowed operations.

`Request` is used when the worker needs a browser or server service the bridge controls.

`Navigate` asks the bridge to update location or submit a navigation request according to the configured routing mode.

### 13.4 DOM operation set

```rust
pub enum DomOp {
    SetText { node: NodeId, text: String },
    SetAttr { node: NodeId, name: AttrName, value: String },
    RemoveAttr { node: NodeId, name: AttrName },
    AddClass { node: NodeId, class: ClassName },
    RemoveClass { node: NodeId, class: ClassName },
    ToggleClass { node: NodeId, class: ClassName, enabled: bool },
    SetStyleVar { node: NodeId, name: CssVarName, value: String },
    SetHidden { node: NodeId, hidden: bool },
    SetValue { node: NodeId, value: String },
    SetChecked { node: NodeId, checked: bool },
    Focus { node: NodeId },
    Blur { node: NodeId },
    ScrollIntoView { node: NodeId, block: ScrollBlock },
    SetScroll { node: NodeId, x: f64, y: f64 },
    PushHistory { url: String },
    ReplaceHistory { url: String },
    Announce { politeness: AriaPoliteness, text: String },
}
```

Rules:

- Operations address generated numeric node ids, not arbitrary selector strings.
- Attribute names, class names, and CSS variable names are validated against generated allow lists.
- Raw `innerHTML` is not part of the default protocol.
- If trusted HTML insertion is required later, it must be a separate privileged operation restricted to renderer-owned sanitized HTML.
- The bridge applies operations in order and should batch layout-affecting work to avoid excessive reflow.
- The bridge reports failed operations in diagnostics mode.

### 13.5 Worker requests

```rust
pub enum WorkerRequestKind {
    FetchServerAction,
    ReadLocalStorage,
    WriteLocalStorage,
    RemoveLocalStorage,
    ReadSessionStorage,
    WriteSessionStorage,
    ClipboardWriteText,
    ClipboardReadText,
    CurrentLocation,
    DocumentVisibility,
    MatchMedia,
}
```

Browser APIs that require permissions or security checks remain brokered by the main-thread bridge. The worker cannot bypass browser restrictions.

## 14. WASM islands

A WASM island is the right model when an area needs real Fission browser execution rather than DOM enhancement.

Good island use cases:

- product configurator;
- cart drawer with complex state;
- address autocomplete;
- rich editor;
- chart explorer;
- live dashboard panel;
- media editor;
- map or 3D control surface;
- authenticated account widget inside a mostly public page.

A route can declare islands explicitly.

```rust
ProductGallery::new(product_id)
    .as_island("product-gallery")

CartDrawer::new()
    .as_island("cart-drawer")
```

An island declaration means:

- server renders initial HTML fallback or loading shell;
- server emits an island mount descriptor;
- CLI compiles a focused WASM artifact for that island entry;
- browser loads only the island artifacts used by the page;
- island communicates with the server through normal Fission actions/jobs/services/capabilities exposed for that island.

The initial implementation may use canvas-rendered Fission islands because that matches the current web shell. A later implementation can support DOM-backed islands if the web renderer grows that capability. The server architecture should not assume every island is canvas forever, but it can start there.

## 15. Island and worker compilation

The CLI should generate temporary build shims under `target/fission/generated/` for each island and worker entry.

```text
target/fission/generated/
  islands/product-gallery/Cargo.toml
  islands/product-gallery/src/lib.rs
  workers/header-nav/Cargo.toml
  workers/header-nav/src/lib.rs
```

Each shim imports the user's application crate and exposes exactly one entrypoint. Normal Rust compilation and dead-code elimination then remove unrelated application code where possible.

Build rules:

- one WASM artifact per island entry by default;
- one WASM artifact per worker entry by default;
- deterministic artifact names with content hashes;
- shared runtime chunks may be introduced later if they improve total payload size without forcing every page to load every island;
- generated shims are build artifacts and must not be checked into the user's source tree;
- source maps and debug metadata are enabled in development and stripped or externalized in release according to config.

The build system must report per-artifact size, compressed size, dependencies, and route usage.

```text
fission site build --project-dir . --release
  workers/header-nav.wasm       18 KB br   used by 12 routes
  workers/search-box.wasm       31 KB br   used by 8 routes
  islands/cart-drawer.wasm     146 KB br   used by 44 routes
  islands/configurator.wasm    284 KB br   used by 3 routes
```

## 16. HTML, CSS, and assets

Server and static HTML rendering should share the same renderer.

The renderer should:

- produce semantic HTML for normal content;
- extract shared CSS classes where possible;
- include route-local critical CSS when configured;
- include generated CSS from design-system tokens;
- include user-provided CSS overrides where configured;
- emit metadata, canonical URLs, Open Graph, structured data, sitemap participation, and robots directives;
- emit only the worker bridge and island loader scripts required by the route;
- preserve accessibility semantics from the Fission tree;
- fail clearly when a widget cannot be lowered for the target.

The page should not ship a global Fission browser runtime unless the route declares a full client app or island that requires it.

## 17. Security model

The server shell must avoid client-controlled execution.

Required rules:

- The browser cannot choose a Rust function to call by string, path, id, or payload.
- The browser cannot instantiate arbitrary widgets.
- The browser cannot request arbitrary jobs or services.
- Server action endpoints require signed action tokens generated by the server.
- Mutating requests require CSRF protection unless an equivalent origin-bound mechanism is configured.
- Route parameters and query fields are validated before reaching route state.
- Cache scopes are enforced by the server shell.
- Private data must not be written to public cache layers.
- Worker DOM operations are restricted to generated node ids and allow-listed attributes/classes/style variables.
- Island/server communication uses registered Fission actions, jobs, services, and capabilities.
- Raw HTML insertion is disallowed by default.
- Debug endpoints must be disabled or authenticated in production.

A secure default ecommerce deployment should be possible without every application author manually designing these boundaries.

## 18. Configuration

Server configuration belongs in `fission.toml`, with bulky environment-specific secrets referenced by environment variables or credential providers rather than stored inline.

```toml
[server]
entry = "crate::server_app"
default_locale = "en-US"
default_route_mode = "revalidated"
render_pass_limit = 4

[server.http]
base_url = "https://shop.example.com"
trust_proxy_headers = true

[server.cache]
provider = "pipeline"

[[server.cache.layers]]
name = "hot"
provider = "moka"
max_capacity = 50000
ttl = "2m"
stale_while_revalidate = "30s"

[[server.cache.layers]]
name = "shared"
provider = "redis"
url_env = "FISSION_REDIS_URL"
prefix = "shop:prod"
ttl = "10m"
stale_while_revalidate = "2m"

[server.sessions]
provider = "cookie"
signing_key_env = "FISSION_SESSION_KEY"
secure = true
same_site = "lax"

[server.workers]
separate_artifacts = true
bridge = "generated"

[server.islands]
separate_artifacts = true
preload = "route"
```

Route-level configuration may live in Rust because route policies need to be close to route declarations and typed application code.

```rust
WebRoute::new("/products/:slug", ProductPage)
    .mode(WebRouteMode::Revalidated(
        RevalidationPolicy::new()
            .ttl(Duration::from_secs(300))
            .stale_while_revalidate(Duration::from_secs(60))
            .tag_from_param("product", "slug")
            .tag("catalog"),
    ))
```

The CLI should validate that `fission.toml` and Rust route metadata agree. If a config file declares Redis but the binary was built without Redis support, readiness checks must fail with an actionable message.

## 19. Developer experience

A developer should be able to start with normal Fission widgets and add server behavior incrementally.

Minimal server app:

```rust
pub fn server_app() -> impl FissionServerApp {
    FissionServerApp::new(StoreApp)
        .routes(store_routes())
}
```

Axum integration:

```rust
let fission = server_app()
    .with_cache(cache)
    .with_session_store(session_store)
    .into_axum_service();

let app = axum::Router::new().nest_service("/", fission);
```

A page starts as a widget:

```rust
pub struct ProductPage {
    pub slug: String,
}

impl Widget<StoreState> for ProductPage {
    fn build(&self, ctx: &mut BuildCtx<StoreState>) -> View {
        let product = ctx.job(ProductDetailsJob { slug: self.slug.clone() })
            .server_policy(ServerResourcePolicy::Blocking)
            .cache_tags([format!("product:{}", self.slug)]);

        FutureBuilder::new(product, |product| ProductDetailsView { product })
    }
}
```

A small DOM enhancement uses a worker:

```rust
SearchFilters::new()
    .enhance_with(SearchFiltersWorker::new())
```

A complex interactive section uses an island:

```rust
ProductConfigurator::new(product_id)
    .as_island("product-configurator")
```

The command surface should expose the target directly:

```text
fission add-target server --project-dir .
fission server check --project-dir .
fission server build --project-dir . --release
fission server serve --project-dir .
fission server routes --project-dir .
fission server cache inspect --project-dir .
fission server cache invalidate --project-dir . --tag product:shoe-123
```

`fission site build` remains the static artifact command. `fission server build` prepares a server binary and web assets that can revalidate pages at runtime.

## 20. Ecommerce example flow

Product page route:

- mode: `Revalidated`;
- TTL: five minutes;
- stale window: one minute;
- tags: `product:{slug}`, `category:{category}`, `catalog`;
- blocking jobs: product details, visible price, availability summary;
- deferred jobs: recommendations, recently viewed products;
- workers: mobile navigation, filter drawer, image thumbnail controls if DOM-only;
- islands: cart drawer, product customizer, address estimator if they need full Fission state.

Cart page route:

- mode: `ServerPrivate`;
- scope: session or user;
- TTL: none or very short private cache;
- actions: update quantity, remove item, apply coupon;
- jobs: shipping methods, tax estimate;
- services: payment status, inventory reservation.

Admin inventory route:

- mode: `ServerPrivate`;
- scope: authenticated user and tenant;
- actions: update stock, publish product;
- services: import status, live stock feed;
- invalidation: product/category/catalog tags after mutations.

Marketing landing page route:

- mode: `Revalidated` or `Static`;
- TTL: long;
- workers: navigation, theme switch;
- islands: none unless it embeds an interactive configurator.

## 21. Observability and diagnostics

The server shell should expose diagnostics suitable for local development and production monitoring.

Required metrics:

- route render count;
- route render duration;
- cache hit/miss/stale hit by layer;
- cache invalidation count by tag;
- job execution duration;
- job cache hit/miss;
- worker artifact sizes;
- island artifact sizes;
- server action success/failure;
- render pass count;
- render failures by route;
- cache privacy violations prevented.

Required development tools:

```text
fission server routes --project-dir .
fission server explain-route --project-dir . --path /products/shoe-123
fission server cache inspect --project-dir . --key <key>
fission server cache invalidate --project-dir . --tag catalog
fission server assets --project-dir . --route /products/shoe-123
```

`explain-route` should show route mode, cache policy, cache key inputs, tags, blocking jobs, deferred jobs, worker artifacts, island artifacts, metadata, and any unsupported widget diagnostics.

## 22. Testing requirements

Unit tests:

- cache key derivation;
- cache scope enforcement;
- Moka TTL and stale-window behavior;
- tag invalidation;
- cache pipeline read-through and promotion;
- worker protocol encoding/decoding;
- DOM operation validation;
- route policy validation;
- action token signing and verification;
- render pass loop limit.

Integration tests:

- revalidated page renders, caches, expires, and rebuilds without recompilation;
- stale-while-revalidate serves stale HTML and refreshes in background;
- Redis implementation passes the same cache contract tests when enabled;
- private route cannot be written to public cache;
- POST action dispatches a real reducer and redirects/rerenders correctly;
- blocking job resolves before HTML output;
- deferred job emits placeholder and enhancement metadata;
- worker artifact is generated only for routes that use it;
- island artifact is generated only for routes that use it;
- generated page remains readable with JavaScript disabled;
- generated worker bridge rejects unauthorized DOM operations.

End-to-end tests:

- ecommerce-like product page uses revalidation and tag invalidation;
- cart page uses private state and server actions;
- route with worker enhancement remains usable without worker loaded;
- route with a WASM island mounts only that island's artifact;
- `fission server serve` can be tested through a real browser.

## 23. Acceptance criteria

The RFC is implemented when all of the following are true:

- A Fission app can expose server routes through at least one Rust HTTP framework adapter.
- A route can be rendered to HTML from a normal Fission widget tree at request time.
- A route can be configured as revalidated with TTL, stale window, and tags.
- Expired pages can be rebuilt by the running server without recompiling or redeploying.
- Moka is available as the default in-memory cache implementation.
- The server shell depends on a `Cache` trait, not directly on Moka.
- A cache pipeline can be configured with at least two layers.
- Cache scope prevents private data from being written to public cache layers.
- Jobs can run as blocking server data dependencies during rendering.
- Actions/reducers can handle POST mutations through signed action tokens.
- A worker can be compiled as a separate WASM artifact and enhance DOM behavior through the generated bridge.
- A WASM island can be compiled as a separate artifact and mounted into a server-rendered page.
- Generated pages include primary content in HTML before JavaScript runs.
- Readiness checks explain missing server, cache, worker, island, or environment configuration.
- Tests cover cache behavior, action dispatch, worker protocol, island artifact generation, and server rendering.

## 24. Implementation sequence

1. Extract a shared HTML/CSS/metadata renderer usable by both static-site and server shells.
2. Add `fission-shell-server` with a minimal request-to-render adapter and one HTTP framework integration.
3. Define route mode, server render policy, revalidation policy, cache key, cache tag, and cache scope types.
4. Define the `Cache` trait and implement Moka.
5. Add route-level full-page caching with TTL and synchronous rebuild on miss/expiry.
6. Add stale-while-revalidate and background rebuild.
7. Add tag indexes and invalidation APIs.
8. Add cache pipeline support and promotion/write-through rules.
9. Add Redis behind an optional feature and run the cache contract tests against it.
10. Add server job drain for blocking jobs and job result caching.
11. Add HTTP action endpoints using existing action/reducer machinery, signed tokens, and CSRF protection.
12. Add worker declarations, stable node ids for worker-controlled DOM, and the generated main-thread bridge.
13. Add worker protocol types, bridge validation, and one DOM enhancement example.
14. Add island declarations and per-island WASM shim generation.
15. Add route asset manifests tying pages to required worker and island artifacts.
16. Add diagnostics commands and readiness checks.
17. Add ecommerce-style example covering revalidated product pages, private cart, workers, and islands.
18. Add full unit, integration, and browser tests.

## 25. Consistency rules

- Revalidated pages are not permanent static files; they are cached render results with TTLs and invalidation.
- Static hosting alone cannot rebuild expired pages; the server shell or another runtime integration is required.
- Jobs are the server data-fetching primitive.
- Services are the long-running and streaming work primitive.
- Actions and reducers are the mutation primitive.
- Workers are for DOM/CSS/window/document progressive enhancement.
- WASM islands are for targeted Fission browser runtimes.
- Full client app mode is for application surfaces, not normal content pages.
- Cache behavior is behind a trait; Moka is the default implementation, not the architecture.
- Redis and future stores are implementations of the cache contract, not new rendering models.
- Browser messages cannot select arbitrary server functions.
- Fission route rendering remains widget-tree based from end to end.
