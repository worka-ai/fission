# Platform capabilities

A Fission app can run in several hosts, but the shared app code should not directly call Android, iOS, browser, or desktop APIs. Those APIs have different permission systems, different lifecycles, and different failure modes. Fission keeps that boundary explicit with capabilities.

A capability is a typed request from your reducer to the host. The reducer says what the app wants to do. The shell host decides whether the current platform supports that request, performs the platform-specific work, and sends a typed result back into your normal action flow.

That separation is what lets one app say "scan a barcode" or "capture audio" without hard-coding an Android activity, an iOS framework object, or a browser JavaScript API into shared state and reducers.

## Add the platform declaration first

Some capabilities require host configuration before the operating system will allow the app to use them. On Android that usually means entries in `platforms/android/AndroidManifest.xml`. On iOS that usually means usage strings in `platforms/ios/Info.plist`, and sometimes entitlements in `platforms/ios/Entitlements.plist`.

Use the command-line interface so the project manifest and generated host files stay in sync:

```sh
fission add-capability camera microphone geolocation --project-dir .
```

The command records the capabilities in `fission.toml` and updates generated platform config where Fission can do that safely. If a platform requires manual setup, app-store configuration, or a paid developer entitlement, the generated README and this guide call that out instead of hiding it.

Desktop targets are part of the same model. Windows and macOS expose many of the same device features that Android and iOS do, while Linux support often depends on the desktop environment, portals, and installed services. Fission records the capability once in `fission.toml`; mobile scaffolds update their platform files immediately, and macOS packaging uses the same capability list to write usage descriptions into the packaged app's `Contents/Info.plist` where the operating system expects them.

## Current built-in capabilities

| Capability | Use it for | CLI value | Mobile config | Desktop and web notes |
| --- | --- | --- | --- | --- |
| [Notifications](/docs/capabilities/notifications/) | Permission checks, local notification display, cancellation, badge updates, push registration hooks | Built into shell setup | App-specific categories, push identifiers, and permission setup are host/provider-owned | Windows and macOS have native notification systems, Linux depends on the desktop notification stack, web usually needs browser permission and a service worker for push |
| [Deep links](/docs/capabilities/deep-links/) | Cold-start and runtime link delivery into typed actions | Built into shell setup | Android intent filters, iOS URL schemes, and associated domains are product-route-specific | Windows and macOS can register schemes during packaging, Linux support depends on desktop registration, web/static sites use real generated routes |
| [NFC](/docs/capabilities/nfc/) | Read/write/emulate NFC tags where the host supports it | `nfc` | Android NFC permission and feature, iOS usage description and NFC entitlement | Desktop support usually requires specific reader hardware or smart-card stacks; web NFC is limited to a small browser/device set |
| [Biometrics](/docs/capabilities/biometrics/) | Ask the host to authenticate the current local user | `biometric` | Android biometric permissions, iOS Face ID usage description | Windows and macOS support local authentication through host providers; web fingerprint prompts are normally passkeys/WebAuthn, not direct biometric access |
| [Passkeys](/docs/capabilities/passkeys/) | Register and authenticate WebAuthn credentials | `passkeys` | Domain association is product-specific; no safe generated defaults | Web uses browser credential APIs; Windows, macOS, Android, and iOS have platform credential APIs; Linux is provider/browser dependent |
| [Barcode scanner](/docs/capabilities/barcode-scanner/) | Scan or decode QR/barcode content | `barcode-scanner` | Camera permission and usage text | Desktop and web can scan through camera providers or decode existing images; Linux camera behavior depends on the media stack |
| [Camera and flashlight](/docs/capabilities/camera/) | Camera availability, permission, photo capture, torch control | `camera` | Android camera permission/features, iOS camera usage description | Windows, macOS, and web have camera APIs; Linux depends on camera devices, portals, and desktop permissions |
| [Clipboard](/docs/capabilities/clipboard/) | Read/write text and typed clipboard content | none required | none generated | Broadly supported on desktop and web, but web browsers often require secure origins and user gestures |
| [Geolocation](/docs/capabilities/geolocation/) | Permission checks and current-position requests | `geolocation` | Android coarse/fine location, iOS when-in-use location usage description | Windows, macOS, and web can expose location services; Linux support varies by desktop service and hardware |
| [Haptics](/docs/capabilities/haptics/) | Impact, notification, selection, and pattern feedback | `haptics` | Android vibration permission, no iOS plist key | Primarily mobile; desktop support is hardware-specific, such as controller rumble or specialized devices |
| [Microphone](/docs/capabilities/microphone/) | Permission checks and bounded audio capture | `microphone` | Android audio recording permission, iOS microphone usage description | Windows, macOS, and web have microphone APIs; Linux depends on audio server and portal configuration |
| [Bluetooth](/docs/capabilities/bluetooth/) | Scan, connect, read/write characteristics, advertise | `bluetooth` | Android Bluetooth permissions/features, iOS Bluetooth usage description | Windows and macOS have native Bluetooth stacks, Linux commonly uses BlueZ, web support is browser-limited |
| [Wi-Fi](/docs/capabilities/wifi/) | Availability, scan, connect, disconnect | `wifi` | Android Wi-Fi/network/nearby-device permissions, iOS Wi-Fi entitlements and location usage text | Desktop Wi-Fi support is provider-driven; web has no general Wi-Fi scan/connect API |
| [Volume control](/docs/capabilities/volume-control/) | Query, set, and adjust host volume streams | `volume-control` | Android audio settings permission, no general iOS system-volume control | Windows, macOS, and Linux can map to host mixer APIs; web should use media element volume rather than system volume |

For a compact cross-platform view, use the [Capability matrix](/docs/capabilities/matrix/).

## Request work from a reducer

Capabilities are normally emitted from reducers because reducers respond to user intent. A button press might dispatch `TakePhoto`, and that reducer asks the host for a camera capture.

```rust
use fission::prelude::*;

#[fission_action]
pub struct TakePhoto;

#[fission_action]
pub struct PhotoCaptured;

fn on_take_photo(
    _state: &mut AppState,
    _action: TakePhoto,
    ctx: &mut ReducerContext<AppState>,
) {
    ctx.effects
        .camera()
        .capture_photo(CameraCaptureRequest {
            facing: CameraFacing::Back,
            flash: CameraFlashMode::Auto,
            ..Default::default()
        })
        .on_ok(ctx.effects.bind(PhotoCaptured, reduce_with!(on_photo_captured)));
}
```

The reducer does not open the camera itself. It records an effect. After the reducer returns, the shell-owned async host performs the request and dispatches the bound result action.

## Register a provider when the default host is not enough

Fission exposes typed provider traits from the shell crates through the `fission` facade. Tests and custom hosts can install memory providers directly:

```rust
use fission::prelude::*;

let app = DesktopApp::new(MyApp)
    .with_camera_host(MemoryCameraHost::default())
    .with_microphone_host(MemoryMicrophoneHost::default())
    .with_bluetooth_host(MemoryBluetoothHost::default());
```

Production shells should provide real platform implementations. If a provider is not registered, the default host reports unsupported errors instead of pretending the platform work succeeded.

## Platform config is part of the feature

Do not treat manifest or plist files as incidental. Operating systems use them to decide whether a feature is visible to the app, whether a permission prompt can be shown, and what text the user sees when the prompt appears.

For generated projects, prefer `fission add-capability ...` over hand-editing platform files. Hand edits are still useful for product-specific text, app-specific deep link routes, associated domains, notification categories, store entitlements, or enterprise deployment requirements. The generated entries are a correct starting point, not a substitute for reviewing platform policy before shipping.

## When a capability is not the right tool

Use a capability when the host owns the work. Use a normal job or service when the work is app-owned Rust logic. Use a widget or overlay when the result is normal in-app UI.

For example, an in-app confirmation dialog should be a Fission widget. A native biometric prompt should be a capability. A long-running sync loop should be a service. A one-shot HTTP request owned by your app should usually be a job.
