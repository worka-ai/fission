# Notifications

Notifications let an app ask the host to show local notifications, schedule reminders, update an app badge, or register for push delivery without embedding platform notification APIs in shared reducers.

Use notifications when the user should receive information outside the currently visible Fission view: a completed export, a missed message, a background sync result, or a scheduled reminder. Keep in-app feedback as Fission UI with `Toast`, `Modal`, or your own overlay; use the notification capability when the operating system or browser owns delivery.

## Setup

This capability is configured as part of shell setup because the required routes, categories, domains, service workers, or provider identifiers are product-specific. Notifications are part of shell setup because categories, push certificates, service workers, URL routes, and app identifiers are product-specific. The CLI does not invent those values. Add the provider in the shell and configure native notification categories, push registration, or browser service-worker files as required by the target you ship.

## Request it from a reducer

Reducers should request notifications work through `ctx.effects`. The reducer records what the app wants, returns control to the runtime, and then receives a typed success or error action when the host finishes.

```rust
ctx.effects
    .notifications()
    .show(NotificationRequest {
        id: NotificationId::new("sync-complete"),
        title: "Sync complete".into(),
        body: "Your workspace is up to date.".into(),
        ..Default::default()
    })
    .on_ok(ctx.effects.bind(NotificationShown, reduce_with!(on_notification_shown)))
    .on_err(ctx.effects.bind(NotificationFailed, reduce_with!(on_notification_failed)));
```

The important rule is that the reducer does not call the operating system directly. It emits a capability effect. That keeps the same app logic usable on desktop, web, Android, iOS, and test hosts even when the real platform implementation is different.

## Platform expectations

Android and iOS have mature notification systems but require target-specific permission prompts and, for push, store/provider registration. Windows and macOS also support native notifications and badges through their desktop notification systems. Linux support depends on the desktop environment and notification daemon. Web notifications work in supporting browsers and usually require a service worker for push.

Read the [capability matrix](/docs/capabilities/matrix/) when you need a quick cross-platform view. The matrix separates platforms with direct support from platforms where a host provider can be implemented but permissions, hardware, or distribution packaging change the behavior.

## Testing

Use `MemoryNotificationHost` to verify reducers emit the right request and handle receipt or error actions. Integration tests should cover permission denial, scheduled notification ids, response actions, and deep links opened from notifications.

For exact request and response types, provider traits, and generated platform configuration, see the [Notifications reference](/reference/capabilities/notifications/).
