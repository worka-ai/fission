# Camera and flashlight

The camera capability asks the host for camera availability, permission, still photo capture, and flashlight control where the active device exposes a torch.

Use camera capture for profile photos, receipts, inspection evidence, document capture, or device setup flows. Use flashlight control as a companion to camera workflows where the host camera has a controllable torch. Do not keep camera capture hidden behind a generic button; make the screen explain why the app needs the camera before the platform prompt appears.

## Setup

Start by declaring the capability with `fission add-capability camera --project-dir .`. Run `fission add-capability camera --project-dir .`. Android receives camera permission plus optional camera/front/flash feature declarations. iOS receives `NSCameraUsageDescription`. Desktop targets can support cameras through host providers; macOS packaged apps need camera usage text in their app bundle, and Windows packaged apps may need webcam capability metadata.

## Request it from a reducer

Reducers should request camera and flashlight work through `ctx.effects`. The reducer records what the app wants, returns control to the runtime, and then receives a typed success or error action when the host finishes.

```rust
ctx.effects
    .camera()
    .capture_photo(CameraCaptureRequest {
        facing: CameraFacing::Back,
        resolution: Some(CameraResolution { width: 1280, height: 720 }),
        flash: CameraFlashMode::Auto,
        quality: Some(90),
        ..Default::default()
    })
    .on_ok(ctx.effects.bind(PhotoCaptured, reduce_with!(on_photo_captured)))
    .on_err(ctx.effects.bind(PhotoCaptureFailed, reduce_with!(on_photo_capture_failed)));
```

The important rule is that the reducer does not call the operating system directly. It emits a capability effect. That keeps the same app logic usable on desktop, web, Android, iOS, and test hosts even when the real platform implementation is different.

## Platform expectations

Android, iOS, web, Windows, and macOS all have camera APIs, but permission UX and device labels differ. Linux support depends on camera devices, portals, and desktop configuration. Flashlight is common on phones and less common on laptops or desktops.

Read the [capability matrix](/docs/capabilities/matrix/) when you need a quick cross-platform view. The matrix separates platforms with direct support from platforms where a host provider can be implemented but permissions, hardware, or distribution packaging change the behavior.

## Testing

Use `MemoryCameraHost` to return deterministic image bytes and to record flashlight requests. Integration tests should cover no-camera devices, permission denial, cancellation, and missing flashlight support.

For exact request and response types, provider traits, and generated platform configuration, see the [Camera and flashlight reference](/reference/capabilities/camera/).
