# Bluetooth

Bluetooth lets an app discover nearby devices, connect to services, exchange characteristic data, and advertise where the platform allows it.

Use Bluetooth for product workflows involving nearby devices: sensors, medical peripherals, controllers, beacons, device provisioning, or field diagnostics. Keep scans explicit and time-bounded. Nearby-device APIs are sensitive because they can reveal physical presence.

## Setup

Start by declaring the capability with `fission add-capability bluetooth --project-dir .`. Run `fission add-capability bluetooth --project-dir .`. Android receives legacy and Android 12+ Bluetooth permissions plus optional hardware feature declarations. iOS receives a Bluetooth usage string. Desktop support should be added through providers using the platform Bluetooth stack.

## Request it from a reducer

Reducers should request bluetooth work through `ctx.effects`. The reducer records what the app wants, returns control to the runtime, and then receives a typed success or error action when the host finishes.

```rust
ctx.effects
    .bluetooth()
    .scan_devices(BluetoothScanRequest {
        service_uuids: vec!["0000180d-0000-1000-8000-00805f9b34fb".into()],
        timeout_ms: Some(8_000),
        include_paired: true,
        allow_duplicates: false,
    })
    .on_ok(ctx.effects.bind(BluetoothDevicesFound, reduce_with!(on_bluetooth_devices_found)))
    .on_err(ctx.effects.bind(BluetoothScanFailed, reduce_with!(on_bluetooth_scan_failed)));
```

The important rule is that the reducer does not call the operating system directly. It emits a capability effect. That keeps the same app logic usable on desktop, web, Android, iOS, and test hosts even when the real platform implementation is different.

## Platform expectations

Android, iOS, Windows, and macOS all expose Bluetooth APIs with platform-specific permission and background rules. Linux support usually goes through BlueZ and can vary by distribution and desktop session. Web Bluetooth is available only in some browsers and generally focuses on Bluetooth Low Energy with user-mediated device selection.

Read the [capability matrix](/docs/capabilities/matrix/) when you need a quick cross-platform view. The matrix separates platforms with direct support from platforms where a host provider can be implemented but permissions, hardware, or distribution packaging change the behavior.

## Testing

Use `MemoryBluetoothHost` for scan/connect/read/write flows. Include unavailable adapter, permission denial, connection lost, unknown characteristic, and write-without-response cases.

For exact request and response types, provider traits, and generated platform configuration, see the [Bluetooth reference](/reference/capabilities/bluetooth/).
